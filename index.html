PRobot — Project Overview

PRobot — Project Overview

1. Purpose

PRobot is a Windows desktop application for interactive 6-axis robot kinematics. It combines a real-time 3D renderer with a forward/inverse kinematics solver and provides a slider-driven UI that lets an engineer explore the entire reachable workspace of a FANUC X-series arm, detect collisions with an obstacle box, and replay pre-recorded motion scripts.

The application is a self-contained research and demonstration tool — no PLC, no driver, no network protocol. Everything executes in-process.

2. Functional Capabilities

Capability Detail

Forward kinematics (FK)

Six joint-angle sliders (S L U R B T). Changing any slider immediately propagates new joint angles through the kinematic chain and redraws the 3D arm.

Inverse kinematics (IK)

Six TCP sliders (X Y Z Rx Ry Rz). On every change, the RBR solver computes up to eight closed-form solutions. The selected stance index chooses one.

Stance selection

A ListBox shows which of the eight IK branches are reachable (Stance N) and which are not (----). The user or script can switch between valid stances without changing the target pose.

Collision detection

Every robot link has a pre-built OBB tree. After each FK/IK update, all link trees are tested against the obstacle-box OBB tree. Hitting links are flagged on the Mechanism and the box turns red.

Script playback

A plain-text file of (X Y Z Rx Ry Rz) waypoints is loaded and stepped through on a 500 ms timer. The script drives the IK path identically to manual slider use.

Click-to-snap

Clicking a face of the obstacle box in the 3D view computes the nearest face normal, sets the TCP orientation to point into that face, and moves the TCP origin to the clicked point.

3.1. Data Flow — IK Interaction

User moves IK slider
        │
        ▼
  mX/mY/mZ/mRx/mRy/mRz updated
        │
        ▼
  ComputeIK()
  ├─ Build target CoordSystem from Euler angles + position
  ├─ RBRSolver.ComputeStances(org, vecZ, vecX)
  │    └─ fills Solutions[0..7]
  ├─ Apply selected stance angles → mJoints[i].JValue
  ├─ mGripper.Xfm ← mTip.Xfm  (gripper follows TCP)
  └─ CheckCollisions()
       ├─ transform box OBB to world space
       ├─ for each link OBB (world space) → OBBCollider.Check()
       ├─ m.IsColliding = hit
       └─ mBoxVN.Color = anyHit ? Red : Blue
        │
        ▼
  Lux scene graph redraws on next vsync
  ├─ MechanismVN   renders arm (colliding links highlighted)
  ├─ TcpVN         draws TCP arrows + origin ring
  └─ InfoVN        prints joint angles + TCP numbers

3.2. Data Flow — FK Interaction

User moves FK slider
        │
        ▼
  m.JValue = sliderValue   (Mechanism joint updated)
        │
        ▼
  OnFK()
  ├─ mGripper.Xfm ← mTip.Xfm
  └─ CheckCollisions()

6. Build and Run

6.1. Prerequisites

  • .NET 10.0 SDK with Windows Desktop workload

  • Visual Studio 2022+ (recommended) — or CLI with MSBuild

  • N:\ drive mounted (Nori sources + robot data)

6.2. Build

dotnet build PRobot/PRobot.csproj
# Output: Bin\PRobot.exe (and all dependent DLLs)

6.3. Run

Bin\PRobot.exe

Or through Visual Studio: F5.

Joint limits (JMin, JMax) are read from the mechanism file and passed directly to RBRSolver. The solver uses them to filter which of the eight algebraic solutions are physically reachable.

8. Kinematics Solver

RBRSolver implements a closed-form analytical inverse kinematics algorithm for the RBR (Roll-Bend-Roll) wrist topology common to 6-axis industrial robots.

Constructor parameters
new RBRSolver(
    a12: 150,    // link offset, joint 1-2 (mm)
    a23: 770,    // link length, joint 2-3 (mm)
    a34: 0,      // link offset, joint 3-4 (mm)
    s2:  0,      // shoulder offset (mm)
    s4:  1016,   // forearm length (mm)
    s6:  175,    // wrist offset to tip (mm)
    min: mMin,   // per-joint minimum angles (rad or deg)
    max: mMax    // per-joint maximum angles (rad or deg)
)

ComputeStances is called with the target TCP origin, approach vector, and orientation vector. It fills Solutions[0..7]. Each Soln carries a flag OK indicating feasibility (within limits, no singularity).

9. Collision Detection

Collision detection uses Oriented Bounding Box (OBB) trees.

Construction (startup)

  • For every Mechanism node that has a CMesh (collision mesh) or a visual Mesh, an OBBTree is built by OBBTree.From(mesh) and stored in mLinkOBBs.

  • The obstacle box OBB is built from the extruded Mesh3 at startup.

Per-frame check

  • BoxWorldXfm computes the current world transform of the box.

  • mBoxOBB.With(BoxWorldXfm) produces a world-space copy of the box OBB tree.

  • For each link, linkOBB.With(m.Xfm) produces a world-space link OBB tree.

  • OBBCollider.Check(linkOBBW, boxOBBW) uses the Separating Axis Theorem (SAT) for an O(log n) narrow-phase check.

Result visualization

  • m.IsColliding = hit — the MechanismVN uses this flag to change the link’s rendered color.

  • mBoxVN.Color = anyHit ? Color4.Red : Color4.Blue

10. Scene Graph

The Lux scene graph (Root) at startup:

GroupVN
├── MechanismVN(mMech)   — renders the full kinematic tree
├── XfmVN(mGripper)      — follows mTip.Xfm; holds a placeholder GroupVN
├── XfmVN(mBoxXfm)       — applies box world transform; holds Mesh3VN(mBoxMesh)
├── TcpVN                — draws TCP axes + ring (Streaming=true)
├── InfoVN               — draws text overlay (Streaming=true)
└── TraceVN.It           — singleton debug print overlay

Streaming = true on TcpVN and InfoVN means their Draw() method is called every frame rather than being cached in a geometry batch.

12. Limitations and Known Issues

  • Windows-only (net10.0-windows, WPF, OpenGL via GLFW).

  • x64-only (FreeType and Nori native binaries are 64-bit).

  • The N:\ drive must be available on every machine that builds the project.

  • There is no .gitignore, so Bin\, obj\, and .vs\ are committed.

  • No automated tests exist; verification is entirely manual.

  • No CI/CD pipeline is configured.
