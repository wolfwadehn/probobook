PRobot — How a Part Is Loaded

PRobot — How a Part Is Loaded

This document traces the full lifecycle of loading and initialising the robot model — from the moment the WPF window opens to the first rendered frame. Two views are provided: a functional (conceptual) view and a line-by-line trace through the source.

1. Functional View

1.1. Phase 1 — Application Bootstrap

The operating system hands control to PRobot.exe. The .NET 10 apphost locates the Desktop runtime, loads PRobot.dll, and calls the WPF entry point. WPF constructs the App object (from App.xaml / App.xaml.cs), which immediately:

  • locks the decimal separator to . on every thread (InvariantCulture), and

  • registers global exception handlers so that any crash shows a dialog rather than a silent process exit.

WPF then instantiates MainWindow as directed by App.xaml’s `StartupUri.

1.2. Phase 2 — Window and OpenGL Surface Creation

MainWindow() does three things in sequence:

  1. Lib.Init() — the Nori framework discovers all loaded assemblies, registers VNode factories, initialises observable streams, and prepares the resource loader (the nori: virtual URI scheme). No GPU work yet.

  2. InitializeComponent() — WPF reads MainWindow.xaml and builds the visual tree: the header bar, the empty Border named mContent (future viewport), and the ScrollViewer + StackPanel (mSettings) on the right.

  3. WPFHost.Init(this, OnLuxReady) — Nori creates an OpenGL-capable Win32 surface via GLFW, wraps it in a WPF UIElement, and stores the OnLuxReady callback. The UIElement is returned and assigned to mContent.Child, embedding the GL surface inside the WPF layout. The OpenGL context is created on a dedicated render thread. OnLuxReady fires once that thread has a live context and has rendered the first (empty) frame.

1.3. Phase 3 — Scene Initialisation (OnLuxReady)

On the first successful GL frame, the Nori render thread posts OnLuxReady back to the WPF dispatcher:

  1. TraceVN.TextColor = Color4.Yellow — debug overlay will print in yellow.

  2. new SceneManipulator() — installs the built-in 3D mouse handler (orbit, pan, zoom).

  3. new RobotScene() — the main loading sequence (Phase 4 below).

  4. Lux.UIScene = scene — the render loop now draws this scene every vsync.

  5. Dispatcher.BeginInvoke(CreateUI) — UI construction is deferred one layout pass so that the panel column has its final width before sliders are sized.

1.4. Phase 4 — Robot Scene Construction (new RobotScene())

This is where the robot part is actually loaded.

1.4.1. 4a. Mechanism Loading

Mechanism.Load("N:/Wad/FanucX/mechanism.curl") reads the CURL file from the N:\ drive. The file encodes a complete kinematic tree:

Root (FanucX)
  └─ Base  (rigid link, no joint)
       └─ S  (Rotate, Z-axis, ±185°)
            └─ L  (Rotate, Y-axis, -180° … +25°)
                 └─ U  (Rotate, Y-axis, -225° … +70°)
                      └─ R  (Rotate, X-axis, ±365°)
                           └─ B  (Rotate, Y-axis, ±120°)
                                └─ T  (Rotate, Z-axis, ±365°)
                                     └─ Tip  (rigid end-effector)

Each node contains:

  • its name, colour, and socket position (where it attaches to its parent),

  • a path to a visual mesh file (e.g., a .msh model of the physical link casting),

  • optionally a separate collision mesh path (lower polygon count),

  • joint type (EJoint.Rotate), rotation axis, and angular limits (JMin, JMax).

Mechanism.Load parses all of this, loads the referenced mesh files from disk, and returns the root Mechanism node with the entire kinematic tree populated.

1.4.2. 4b. Node Lookup

After the tree is in memory, two named descendants are retrieved:

  • Tip — the end-effector node. Its Xfm property always holds the current world-space TCP transform after any joint change.

  • The six joint nodes S, L, U, R, B, T — accessed as an array mJoints[0..5] built from "SLURBT".Select(a ⇒ mMech.FindChild(a.ToString())!).

1.4.3. 4c. Joint Limits

The per-joint limits JMin and JMax are copied into mMin[] and mMax[]. These plain arrays are passed to RBRSolver so the solver can flag which of its eight algebraic branches fall within the physical travel range.

1.4.4. 4d. IK Solver Construction

new RBRSolver(150, 770, 0, 0, 1016, 175, mMin, mMax) creates the closed-form solver. The six numeric parameters are the Denavit-Hartenberg link lengths and offsets of the FANUC X arm:

Parameter Value (mm) Meaning

a12

150

Lateral offset between joint 1 (S) and joint 2 (L)

a23

770

Length of the lower arm (L to U)

a34

0

Offset at the elbow

s2

0

Shoulder lateral offset

s4

1016

Length of the upper arm (U to wrist centre)

s6

175

Distance from wrist centre to TCP

1.4.5. 4e. Obstacle Box

The 200 × 200 × 200 mm box is built entirely in code:

  1. Poly.Rectangle(-100, -100, 100, 100) — a 2D square centred at the origin.

  2. Mesh3.Extrude([boxPoly], 200, Matrix3.Translation(0, 0, -100), ETess.Medium) — extrudes the square 200 mm, offset by −100 mm on Z so the box is centred at its local origin.

  3. OBBTree.From(mBoxMesh) — builds the collision hierarchy for the box.

  4. new Mesh3VN(mBoxMesh) { Mode = EShadeMode.Glass, Color = Color4.Blue } — the visual node.

  5. new XfmVN(BoxWorldXfm, mBoxVN) — wraps the visual node in a transform node; the transform is updated whenever the obstacle sliders change.

1.4.6. 4f. Link OBB Trees

Every node in the mechanism tree is inspected:

foreach (var m in mMech.EnumTree()) {
    var cm = m.CMesh;
    if (cm != null)         mLinkOBBs[m] = OBBTree.From(cm);
    else if (m.Mesh != null) mLinkOBBs[m] = OBBTree.From(m.Mesh);
}

Priority: collision mesh (CMesh) over visual mesh (Mesh). Nodes with neither mesh (e.g., the Tip end-effector or pure axis joints) are skipped. The resulting dictionary mLinkOBBs maps each Mechanism node to its OBB tree, all in local (link) space.

1.4.7. 4g. Gripper Transform Node

new XfmVN(Matrix3.Identity, new GroupVN([])) — a placeholder transform node whose Xfm is updated to mTip.Xfm every frame after FK/IK computation, effectively making it track the TCP. An empty GroupVN child is used because no gripper geometry is loaded in the demo.

1.4.8. 4h. Home Pose and First IK

mCS = mHome sets the initial target coordinate system to the robot’s home pose. ComputeIK() is called immediately, ensuring the solver runs once before the UI appears so mJoints have correct values.

The home coordinate system is:

readonly CoordSystem mHome = new(
    new(1166, 0, 1161 - 565),   // origin at (1166, 0, 596) mm in world space
    Vector3.XAxis,               // X axis of the home frame = world X
    Vector3.YAxis                // Y axis of the home frame = world Y
);

1.4.9. 4i. Script Timer

new DispatcherTimer { Interval = TimeSpan.FromMilliseconds(500) } — creates a WPF timer (runs on the UI thread) for stepping through waypoints. The Tick handler calls TickScript(). The timer is disabled at this point.

1.4.10. 4j. Overlay Visual Nodes

new InfoVN(this) and new TcpVN(this) — the two streaming VNodes that render text and TCP axes every frame. Both are created with a reference to RobotScene so they can read current pose data.

1.4.11. 4k. Scene Graph Assembly

BgrdColor = Color4.Gray(64);          // dark grey background
Bound = new Bound3(-1200,-1200,0, 1200,1200,1500);  // camera auto-fit region
Root = new GroupVN([
    new MechanismVN(mMech),  // full robot arm
    mGripper,                // TCP-tracking transform
    mBoxXfm,                 // obstacle box
    mTcpVN,                  // TCP axes overlay
    mInfoVN,                 // text overlay
    TraceVN.It               // debug print overlay
]);

After this line, the Lux render loop has a complete, valid scene graph. The next vsync produces the first rendered frame showing the robot in its home pose.

1.5. Phase 5 — UI Population (CreateUI)

One WPF layout tick after OnLuxReady, Dispatcher.BeginInvoke fires and calls scene.CreateUI(mSettings.Children). This iterates mMech.EnumTree() again and builds one WPF Slider per joint. The FK sliders read their initial value from the joint’s current JValue (set by Phase 4h).

The IK sliders are initialised from mX/mY/mZ/mRx/mRy/mRz, which are 0.0 except mRx (set to −90 inside AddSlider as a special case to match the initial tool-up orientation).

1.6. Phase 6 — Steady-State Operation

From this point the application is event-driven:

  • Slider moved → FK or IK callback → kinematic update → CheckCollisions() → Lux redraws on next vsync.

  • Viewport click → Lux pick system → Picked(obj) → SnapToFace if box was clicked.

  • Load + Play → LoadScript(path) populates mScript → TogglePlay() starts the timer → TickScript() advances waypoints every 500 ms.

2. Line-by-Line Trace

This section walks through every relevant source line in execution order, with the source location and a brief explanation of each statement’s effect.

2.1. App.xaml.cs — Application Startup

// L15
Thread.CurrentThread.CurrentCulture = CultureInfo.InvariantCulture;

Forces the main thread to use . as the decimal separator. Critical for double.Parse in script loading.

// L16
CultureInfo.DefaultThreadCurrentCulture = CultureInfo.InvariantCulture;

Propagates the same culture to all threads spawned from this point.

// L17
AppDomain.CurrentDomain.UnhandledException += OnDomainUnhandledException;

Catch-all for non-dispatcher thread exceptions.

// L18
DispatcherUnhandledException += OnUnhandledException;

Catch-all for the WPF message loop.

2.2. MainWindow.xaml.cs — Window Construction

// L13
Lib.Init();

Entry point into the Nori framework. Scans loaded assemblies, registers VNode factories (so VNode.MakeFor can create the right VNode subclass for any domain type), initialises observable streams used by the reactive pattern, and sets up the nori: URI resolver. Must be the first Nori call.

// L14 (first call)
InitializeComponent();

Standard WPF: parses MainWindow.xaml, constructs the full visual tree including mContent (Border) and mSettings (StackPanel). After this line, the WPF layout is valid but not yet displayed.

// L14 (second call)
mContent.Child = WPFHost.Init(this, OnLuxReady);

WPFHost.Init creates a GLFW window attached as a child of the WPF window, creates an OpenGL 4.x core context on a dedicated thread, and returns a WPF HwndHost (or equivalent) that occupies mContent. The returned element is immediately parented. OnLuxReady is stored as a callback; it will fire from the render thread once the first GL frame completes, marshalled back to the WPF dispatcher.

2.3. MainWindow.xaml.cs — OnLuxReady

// L19
TraceVN.TextColor = Color4.Yellow;

Configures the debug overlay colour before any Lib.Trace calls might occur during scene init.

// L20
new SceneManipulator();

Installs the built-in Nori 3D mouse handler. Subscribes to mouse events from Hub.Mouse and translates drag gestures into camera transforms on Lux.UIScene.

// L21
var scene = new RobotScene();

Triggers the full part-loading sequence. See the RobotScene() walkthrough below.

// L22
Lux.UIScene = scene;

Sets the active scene. The Lux render loop immediately begins drawing scene on every vsync. At this point the robot arm is visible (home pose, no collision).

// L24
Dispatcher.BeginInvoke(() => scene.CreateUI(mSettings.Children));

Defers UI population. BeginInvoke queues the lambda at DispatcherPriority.Normal, which runs after the current layout pass finishes. This ensures mSettings has been measured and arranged by the time sliders are added.

2.4. RobotScene.cs — Constructor, line by line

public RobotScene() {

Begins construction. No base-class arguments needed; Scene3 has a default constructor.

    mMech = Mechanism.Load("N:/Wad/FanucX/mechanism.curl");

The central part-loading call. Mechanism.Load:

  1. Opens the .curl file from the N:\ drive.

  2. Parses the tree structure, reading each node’s name, joint type, limits, mesh references, and socket position.

  3. For each node that has a mesh path, loads the mesh file (likely .msh binary format) from disk, constructing a Mesh3 with Node[] (position + normal) and triangle indices.

  4. Returns the root Mechanism — the entire robot arm tree in memory, meshes and all.

If the file or any referenced mesh is missing, an exception propagates to the global handler.

    mTip = mMech.FindChild("Tip")!;

Recursive search for the "Tip" child. Tip is the end-effector node at the bottom of the chain. ! asserts non-null (the mechanism file is known to contain this node).

    mJoints = [.. "SLURBT".Select(a => mMech.FindChild(a.ToString())!)];

LINQ over the six characters S, L, U, R, B, T. Each FindChild call traverses the tree looking for a node with that single-character name. The result is collected into a Mechanism[] array using the C# 12 collection expression spread ([..]).

    for (int i = 0; i < 6; i++) {
        var m = mJoints[i];
        mMin[i] = m.JMin;
        mMax[i] = m.JMax;
    }

Copies the physical joint limits into plain double arrays. JMin/JMax come directly from the CURL file (e.g., S: −185° … +185°).

    mSolver = new(150, 770, 0, 0, 1016, 175, mMin, mMax);

Constructs the closed-form IK solver. The six numeric parameters encode the FANUC X DH parameters. The solver stores the limits to classify solutions as OK or not.

    Lib.Tessellate = FastTess2D.Process;

Registers the 2D polygon tessellator. Without this, Mesh3.Extrude would have no way to triangulate the cross-section contour.

    var boxPoly = Poly.Rectangle(-100, -100, 100, 100);

Creates a 200 × 200 mm 2D closed rectangle, centred at the origin. This is the cross-section of the obstacle box.

    mBoxMesh = Mesh3.Extrude([boxPoly], 200, Matrix3.Translation(0, 0, -100), ETess.Medium);

Extrudes the rectangle:

  • Height = 200 mm.

  • Matrix3.Translation(0, 0, -100) shifts the base plane down 100 mm so the box is centred at Z = 0.

  • ETess.Medium controls arc tessellation tolerance (no arcs here, so it primarily affects the triangle count of the flat caps).

Result: a closed box mesh with six faces, centred at its local origin.

    mBoxOBB = OBBTree.From(mBoxMesh);

Builds an OBB tree from the box triangles. For a simple box this is trivial — likely one or two levels.

    mBoxVN = new Mesh3VN(mBoxMesh) { Mode = EShadeMode.Glass, Color = Color4.Blue };

Creates a visual node for the box. Glass mode = semi-transparent. Initial colour blue (no collision).

    mBoxXfm = new XfmVN(BoxWorldXfm, mBoxVN);

Wraps the box visual node in a transform node. BoxWorldXfm is a property that returns Matrix3.Translation(mBX, mBY, mBZ). Initial position: (700, 0, 700) mm.

    foreach (var m in mMech.EnumTree()) {
        var cm = m.CMesh;
        if (cm != null)           mLinkOBBs[m] = OBBTree.From(cm);
        else if (m.Mesh != null)  mLinkOBBs[m] = OBBTree.From(m.Mesh);
    }

Iterates every node in the kinematic tree. For each node that has geometry, builds an OBB tree in the node’s local space. This happens once at startup; the world-space transform is applied lazily in CheckCollisions() via OBBTree.With(m.Xfm).

    mGripper = new XfmVN(Matrix3.Identity, new GroupVN([]));

A transform node with an empty child group. Acts as a TCP-following marker in the scene graph. mGripper.Xfm is set to mTip.Xfm after every FK/IK update.

    mCS = mHome; ComputeIK();

Two statements on one line:

  1. mCS = mHome — sets the target coordinate system to the home pose.

  2. ComputeIK() — runs the full IK pipeline (solver, joint angles, collision check) before the first frame. This ensures the robot is in a physically consistent state from the very first draw.

    mPlayTimer = new DispatcherTimer { Interval = TimeSpan.FromMilliseconds(500) };
    mPlayTimer.Tick += (_, _) => TickScript();

Creates the script step timer. DispatcherTimer fires on the WPF dispatcher thread, so TickScript can safely modify UI-related state.

    mInfoVN = new InfoVN(this);
    mTcpVN  = new TcpVN(this);

Creates the two streaming overlay nodes. Both receive a reference to this so they can read live data via InfoData and Tip.Xfm each frame.

    Lib.Tracer  = TraceVN.Print;

Routes all Lib.Trace(…) calls to the viewport overlay, where they appear as yellow text lines.

    BgrdColor   = Color4.Gray(64);

Sets the OpenGL clear colour to dark grey (value 64 out of 255).

    Bound       = new Bound3(-1200, -1200, 0, 1200, 1200, 1500);

Defines the bounding box of the scene in world space. Used by ZoomExtents() and the initial camera fit. The robot’s full workspace is roughly within this 2.4 m × 2.4 m × 1.5 m volume.

    Root = new GroupVN([
        new MechanismVN(mMech),
        mGripper,
        mBoxXfm,
        mTcpVN,
        mInfoVN,
        TraceVN.It
    ]);

The scene graph is now complete. Each element:

  • MechanismVN(mMech) — renders the full robot. It internally traverses mMech.EnumTree(), creates sub-VNodes per link, and uses m.Xfm for positioning. It also reads m.IsColliding to tint colliding links.

  • mGripper — TCP-tracking transform; currently invisible (empty group child).

  • mBoxXfm — obstacle box with its position transform.

  • mTcpVN — TCP coordinate frame arrows.

  • mInfoVN — text overlay.

  • TraceVN.It — debug trace output.

Setting Root completes the Scene3 setup. The Lux render loop, which began running when Lux.UIScene = scene was set, now has a non-null root and will traverse this graph on every vsync.

2.5. RobotScene.cs — ComputeIK() (first call)

This is the initial IK run, invoked from the constructor with mX = mY = mZ = 0 and mRx = mRy = mRz = 0.

void ComputeIK() {
    mComputingIK = true;

Guards against re-entrant calls from the stance ListBox.SelectionChanged handler.

    var cs = CoordSystem.World;
    cs *= Matrix3.Rotation(EAxis.X, mRx.D2R());
    cs *= Matrix3.Rotation(EAxis.Y, mRy.D2R());
    cs *= Matrix3.Rotation(EAxis.Z, mRz.D2R());

Builds a coordinate system by successively applying Euler rotations. D2R() is an extension method converting degrees to radians. On the first call all angles are 0, so cs equals the world identity frame.

    var tcp = (Vector3)(mHome.Org + new Vector3(mX, mY, mZ));
    mCS = cs * Matrix3.Translation(tcp + cs.VecZ * 50);

Computes the TCP world position as home + offset. Adds 50 mm along the approach axis (cs.VecZ) to account for tool length. On the first call: tcp = (1166, 0, 596), offset = 50 mm in world Z.

    mStances.Items.Clear();
    mSolver.ComputeStances(mCS.Org, mCS.VecZ, mCS.VecX);

Clears the ListBox items, then runs the IK solver. The solver fills mSolver.Solutions[0..7] with all eight algebraic solution branches.

    for (int j = 0; j < 8; j++) {
        var a = mSolver.Solutions[j];
        mStances.Items.Add(a.OK ? $"Stance {j + 1}" : "----");
        if (j == mSelStance)
            for (int i = 0; i < 6; i++)
                mJoints[i].JValue = a.GetJointAngle(i);
    }

For each of the eight solutions:

  • Adds "Stance N" or "----" to the ListBox.

  • For the selected stance (mSelStance = 0 initially): writes each joint angle into mJoints[i].JValue. Setting JValue on a Mechanism node immediately recomputes its and all descendant world transforms, so after this loop mTip.Xfm is already updated.

    mGripper.Xfm = mTip.Xfm;

Moves the gripper transform node to the TCP position. On the first call this is effectively a no-op (identity pose), but it must be kept in sync.

    CheckCollisions();

Tests all link OBBs against the box OBB. On the first call, the robot is at home and the box at (700, 0, 700), so no collision is expected.

    mComputingIK = false;
}

Clears the re-entrant guard.

2.6. RobotScene.cs — CheckCollisions() (called from ComputeIK)

void CheckCollisions() {
    var boxOBBW = mBoxOBB.With(BoxWorldXfm);

BoxWorldXfm = Matrix3.Translation(700, 0, 700) on the first call. With(xfm) returns a new OBBTree with all bounding boxes transformed into world space.

    bool anyHit = false;
    using var bc = OBBCollider.Borrow();

Borrows an OBBCollider from the pool. The using pattern ensures it is returned when the method exits.

    foreach (var (m, linkOBB) in mLinkOBBs) {
        bool hit = bc.Check(linkOBB.With(m.Xfm), boxOBBW);
        m.IsColliding = hit;
        if (hit) anyHit = true;
    }

For each robot link: transform its local OBB to world space using m.Xfm (the link’s current world transform, updated by the preceding JValue assignments), then SAT-test it against the box OBB. Set m.IsColliding so MechanismVN can highlight the link.

    mBoxVN.Color = anyHit ? Color4.Red : Color4.Blue;
}

Box turns red if any link collides; stays blue otherwise.

2.7. First Rendered Frame

At this point, the next vsync triggers the Lux render loop:

  1. Clears the framebuffer to Color4.Gray(64).

  2. Traverses the scene graph starting at Root.

  3. MechanismVN.Draw() — renders all link meshes with correct world transforms.

  4. mBoxXfm → Mesh3VN.Draw() — renders the blue semi-transparent box at (700, 0, 700).

  5. TcpVN.Draw() — draws three coloured arrows and a white ring at the TCP.

  6. InfoVN.Draw() — renders four lines of text in the lower-left corner.

  7. TraceVN.It.Draw() — no trace messages yet, draws nothing.

  8. Swaps buffers.

The robot is now visible, at home pose, with the box in the scene and the overlay showing all zeros. The user can interact immediately.
